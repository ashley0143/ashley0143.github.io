Snaps are slow to launch

Snaps creates a virtual device for itself which it uses, this process takes some cycles. Opening a snap after should not take much longer. Snaps being slower to launch is matterless as most packages shipped in Snaps are not constantly cycled for maximum throughput.

Snaps are bloated

Given you only care for possible overhead with dependencies and virtual drives yes, snaps using that comparison are bloated. But that's the wrong system to deploy them on anyway, what snaps offer in return far outweight the minor kicks in memory usage.

Snaps don't share GTK themes

Being able to read a GTK theme from a user is in some cases undesirable for the developer and potentially the user As themes are the, if not the biggest UI breaking factor in any graphical environment which supports it. The user is able to change this hence the overall effect is practically none.

Snaps are proprietary

No part of the snap client is proprietary, this confusion arises from the snapstore being proprietary.

Snap offers no benefits to normal style package managing

No, Snap offers the exact opposite. Snap offers a way of building and shipping packages in a self dependant package format which dosen't interfere with the rest of the system components. This by consequence eliminates one of the bigger problems in package management which is dependency resolving and managing. Packages hence are much less prone to breakage by a system update/etc. Snap offers a reproducible way to build packages.
